Q1 — Hash Adapter Merging
How many collapse?
A hash-adapter pattern unifies any algorithm family where the only real difference is which hash function (or security-level parameter) is plugged in — the construction is identical. The result is a single parameterized entry that accepts a hash config object and dispatches to the correct implementation.

The adapter contract

Each merged family produces one canonical function: algo(config)(input) → output where config is the hash/parameter object. If no config is provided the pre-configured default is used. This is not a wrapper — the inner implementation must receive the hash as a first-class construction param, not a runtime string.

Inventory reduction via merge 156 of 776 → 20.1% reduction
What qualifies for merge

An algo merges only when: (1) the underlying mathematical construction is identical, (2) the only variance is hash function, key size, output width, or security level parameter, and (3) the implementation can be expressed as construct(params) where params contain the variant selector. Block cipher modes (ECB/CBC/GCM) do NOT merge with each other — they have distinct security properties. Nor do AEAD schemes like AES-GCM vs AES-CCM — different constructions entirely.
